* **Respuesta API:** `{"total": 20}`
* **Análisis:** La API devuelve los 20 docentes programados correctamente. **Comportamiento correcto.** La corrección anterior del repositorio está funcionando en vivo. El payload incluye los docentes sin DNI.

## 3. Módulo: Historial XML
* **Endpoint:** `GET /api/schedule/xml-uploads`
* **Realidad DB:** 1
* **Respuesta API:** `{"total": 1}`
* **Análisis:** **Comportamiento correcto.** El historial XML sí está siendo enviado al frontend de forma íntegra.

## 4. Módulo: RPT Planillas
* **Endpoint:** `GET /api/rpt-planilla/docentes`
* **Realidad DB:** Debería consolidar a los 20 docentes y devolver las planillas calculadas.
* **Respuesta API (CRÍTICA):** **Error 500 Internal Server Error.**
* **Payload Devuelto:** `{"success":false,"data":null,"error":"Error interno del servidor. Por favor, contacte al soporte técnico."}`
* **Análisis Profundo:** Tras extraer el traceback interno de FastAPI, se descubrió un `fastapi.exceptions.ResponseValidationError: 15 validation errors`. 
La validación falla explícitamente en el campo `dni` porque FastAPI Pydantic está aplicando el esquema `TeacherHoursItem` el cual exige `dni: str` (estricto) y revienta si recibe `None`.

## Conclusión API
El único endpoint verdaderamente roto a nivel backend es `/api/rpt-planilla/docentes`. El resto de los endpoints **sí están devolviendo la información completa** a la web. Esto señala que hay componentes del Frontend que están descartando silenciosamente la data enviada por los otros módulos.

# Architecture Audit: Global Structure & SPA Lifecycle

**Audit Date:** 2026-05-13
**Status:** Phase 1 (Static Analysis)
**Severity Level:** GREEN / AMBER (Stable but needs modernization)

---

## 1. Backend Architecture (FastAPI)

### Patterns & Structural Health
- **Application Factory Pattern:** The project correctly implements the Factory Pattern (`create_app`), ensuring isolation and preventing global state pollution.
- **Layer Separation:** 
    - **Routers:** Properly handle HTTP contracts and permissions.
    - **Services:** Centralize complex academic logic (Consolidators, XML Parser).
    - **Repositories:** Isolate SQLAlchemy queries.
    - **Status:** **EXCELLENT**.

### Risks & Debt
- **Lifecycle Management [LOW]:** Uses `@application.on_event("startup")`, which is deprecated. Transition to `lifespan` is recommended for future-proofing.
- **Circular Imports [LOW]:** Routers are imported inside `setup_routers` to mitigate circularity. While safe, it indicates high coupling between modules.

---

## 2. Frontend Architecture (SPA ES6)

### Modularity & Lifecycle
- **Modular ES6:** Modules are well-defined (`docentes.js`, `horarios.js`, etc.).
- **Global Event Delegation:** Correctly uses a centralized handler registry (`Handlers`) and delegation to manage dynamic DOM elements.
- **Bootstrap Lifecycle:** 
    - **Hardenened:** Implementation of `watchdog` (10s) to detect initialization deadlocks.
    - **Security:** Session hydration via `checkSession` before module mounting.

### Risks & Debt
- **Navigation Fallback [MEDIUM]:** The fallback routine is robust but relies on an ordered array in `main.js`. If a new module is added without updating this array, it becomes invisible to the fallback logic.
- **UI State Management [LOW]:** State is largely DOM-dependent or stored in `localStorage`. There is no central state store (like Redux/Vuex), which is acceptable for a utility SPA but increases risk as complexity grows.

---

## 3. Modular Boundaries

| Boundary | Status | Risk |
| :--- | :--- | :--- |
| **RBAC <-> UI** | **SECURE** | Permission maps in `api.js` decouple logic. |
| **XML <-> DB** | **STRICT** | Handled by `XMLUploadService` with transactional safety. |
| **RPT <-> Calculations** | **FRAGILE** | High reliance on hardcoded time constants (Recess logic). |

---

## 4. Findings & Recommendations

### [CRITICAL] None Detected
The current architecture is stable and follows the "Global Engineering Rules".

### [MEDIUM] Lifecycle Modernization
> **Impact:** Future FastAPI compatibility.
> **Probability:** High (on next major update).
> **Regression Risk:** Low.
> **Proposal:** Migrate `startup_event` to a `lifespan` context manager in `app/main.py`.

### [LOW] State Consistency
> **Impact:** UI glitches in edge cases.
> **Probability:** Medium.
> **Regression Risk:** Minimal.
> **Proposal:** Standardize `init` return values for all JS modules to ensure the `watchdog` can accurately track initialization.

# 🔍 Informe de Auditoría Técnica: Schedule Management Service

**Fecha:** 21 de Abril de 2026
**Perfil del Auditor:** Senior Software Architect
**Área de Alcance:** Frontend (`prototype`), Backend (FastAPI), Base de Datos y Pipeline de Migraciones.

---

## 1. VISIÓN GENERAL DEL PROYECTO

- **Objetivo del Sistema:** Es una plataforma empresarial de gestión académica diseñada para conciliar la carga teórica (importada desde aSc Horarios vía XML) con la realidad operativa corporativa (incidencias, reemplazos, faltas) para finalmente auditar e inyectar datos en una planilla de pagos.
- **Problema que resuelve:** El divorcio organizativo entre los horarios planificados, las inasistencias reales de los docentes, y el cálculo de la liquidación de horas (quién dictó qué y por ende a quién se le paga). Evita la duplicación de identidades de los docentes (homónimos o mal escritos en XML) mediante *Fuzzy Matching*.
- **Estado Actual:** El proyecto se encuentra en una **fase de transición crítica de desarrollo a pre-producción**. Si bien el *backend* core parece maduro y funcional, existe un cuello de botella arquitectónico severo en el *frontend* y se está ejecutando en pleno vuelo una refactorización estructural profunda de la base de datos (Migración a UUID).

---

## 2. ARQUITECTURA DEL SISTEMA

- **Tipo de Arquitectura Backend:** Modular Monolítica impulsada por una adaptación de *Domain-Driven Design* (DDD) simplificado. Los módulos están bien aislados en carpetas (`docentes`, `horarios`, `observaciones`, `reportes`, etc.) bajo el esquema `router` ➔ `service` ➔ `repository`.
- **Tipo de Arquitectura Frontend:** **Monolito de Archivo Único (SPA SPA_Legacy)**. La lógica UI está fuertemente acoplada en un solo archivo `index.html` gigantesco (aprox. 300KB) usando *Vanilla JS* y IIFEs (Immediately Invoked Function Expressions) ancladas al objeto `window`.
- **Flujo General de Datos:**
  1. Ingesta Massiva (Archivo XML) ➔ Parser ➔ Motor MDM (Master Data Management).
  2. Motor Fuzzy Matching resuelve identidades de docentes (staging `teachers_sinasignar`).
  3. UI consume la carga académica y reporta incidencias (Faltas/Reemplazos).
  4. Lógica de Negocio Binaria (Backend Service) calcula en caliente en memoria las horas reales ➔ Retorna Excel Export o JSON.

---

## 3. STACK TECNOLÓGICO Y JUSTIFICACIÓN

- **Backend:** Python 3.10+, FastAPI, Uvicorn.
  *Justificación:* Excelente para I/O asíncrono y alta concurrencia. FastAPI permite una validación de Pydantic y tipado fuerte, crucial para este tipo de sistema transaccional.
- **Base de Datos / ORM:** PostgreSQL + SQLAlchemy 2.0 + Alembic (Migraciones).
  *Justificación:* Sistema 100% relacional por naturaleza (Docente ➔ Cursos ➔ Sesiones). SQLAlchemy 2 aporta el paradigma de session management moderno y manejo de *joinedloads* activos.
- **Frontend:** Vanilla JS / HTML5 (Sin frameworks).
  *Justificación Operativa:* Presumiblemente concebido como un prototipo rápido que escaló fuera de control. **No es sostenible a largo plazo.**
- **Algoritmos Core:** `rapidfuzz` (o `difflib`).
  *Justificación:* Vital para implementar la similitud sintáctica y juntar "Pérez, Juan" del XML con "Juan Perez" de la base maestra, evitando polución de datos.

---

## 4. FUNCIONALIDADES IMPLEMENTADAS

- **✅ Implementado y Funcional:**
  - Importación y desglosado de Archivos XML.
  - MDM de Docentes: Normalización de nombres, soporte para uniones de identidades duplicadas.
  - Gestión de incidencias (Observaciones): Lógica sólida para suplantar la paga (100%/0%) entre titular y reemplazante.
  - Autenticación y Autorización (JWT + Validaciones RBAC explícitas en cada ruta).
- **⚠️ En Desarrollo / Pendiente:**
  - Migración profunda de identificadores secuenciales enteros a `UUIDv4`.
  - Desacoplamiento del UI Frontend Monolítico (existen scripts como `refactor_modular.py` en curso).
  - Cálculo de horas parametrizables por Sedes (Actualmente hay *hardcoding* de recesos visualizado en reportes).

---

## 5. ERRORES, PROBLEMAS DETECTADOS Y DEUDA TÉCNICA

A continuación, un listado de riesgos de arquitectura graves si el estado actual se lleva a producción:

### 🔴 CRÍTICO: Choque de Modelos por Migración UUID
Se detectó convivencia híbrida en la base de datos (`id: UUID` vs `legacy_id: Integer`). Archivos como `service.py` del módulo docentes tienen mecanismos defensivos pesados (`resolve_teacher`) intentando inferir qué tipo de ID llega de la UI. Esta fricción causa **errores 500 generalizados** si los Pydantic schemas del backend no hacen match 1:1 con las peticiones asíncronas del Front (que suele enviar enteros).

### 🔴 CRÍTICO: Estado del Frontend ("God Page")
El archivo `prototype/index.html` es una bomba de tiempo de ~304KB. El uso de variables globales ancladas a `window.` (e.g., `window.handleObsTypeChange`) provoca *ReferenceErrors* catastróficos por condiciones de carrera o colisiones de *scope* en el navegador. No hay bundler de por medio, ni encapsulamiento por componentes.

### 🟡 ALTO: Fugas de Seguridad en Middleware
En `main.py`, se expone explícitamente el *Traceback real de errores 500* a toda petición (`debug_exception_handler`). Esta práctica de desarrollo jamás debe tocar producción, es una puerta de entrada para ataques al revelar la estructura de archivos local.
Adicionalmente, en `app/dependencies/auth.py`, existe una cláusula *Bypass* absoluta si el rol es 'SISTEMAS', omitiendo por completo los logs transaccionales estrictos o comprobaciones de permisos de auditoría.

### 🟡 MEDIO: Algoritmo de Reportes No-Transaccional
El reporte de pago asume la sustitución y horas en un motor "en memoria" (lógica de capas de servicio descrita en architecture.md). Aunque funciona, cualquier cambio en las reglas de negocio base afectará las lecturas históricas de semanas pasadas, pues los "0.33 hours" restados de recesos no se consolidan físicamente en auditorías.

---

## 6. CALIDAD DEL CÓDIGO (BACKEND)

- **Lo Positivo:** El Backend es altamente disciplinado bajo metodologías de inyección de dependencias y separaciones en 3 capas. Excelentes capturas transaccionales `db.rollback()` en operaciones MDM (ej. al fusionar docentes) aseguran integridad.
- **Lo Negativo:** Uso excesivo de *Monkey Patching* temporal. Hay rutas y scripts marcados con "v3.10" y validaciones defensivas de tipeos en caliente (`isinstance()`) generados por la ausencia de un contrato estricto de API.
- **Riesgo:** Alto nivel de esfuerzo para depurar un error frontend debido a la falta de un sistema de *Linting* como ESLint en Javascript y un tipado fuerte.

---

## 7. DECISIONES DE INGENIERÍA ACTUALES Y TRADE-OFFS

| Decisión | Por qué se hizo | Trade-off (Costo) |
| :--- | :--- | :--- |
| **Vanilla JS "Todo en uno"** | Velocidad inicial e iteración rápida sin dependencias ni compilaciones node.js. | Espagueti code inmanejable. Complejidad exponencial para nuevos *features*. |
| **Normalización String MDM** | El XML fuente no provee DNI garantizado, obligando a vincular por nombre. | Riesgo real de homónimos puros fusionados erróneamente sin intervención manual. |
| **UUID Migration en caliente** | Incrementar nivel de seguridad ID y descentralizar generación de *Primary Keys*. | Costo alto en horas hombre parchando serializadores y endpoints rotos por conflicto de `int` a `str`. |

---

## 8. ESTADO ACTUAL REAL DEL PROYECTO

El sistema posee un **nivel de madurez de Prototipo Avanzado / Beta**. Tiene toda la inteligencia algorítmica detrás, pero carece de un cascarón visual sustentable y herramientas de monitoreo en producción.
**¿Qué tan cerca está de ser estable/desplegable?**
Todavía faltan esfuerzos de estabilización en torno a los serializadores y la arquitectura del front, que no superaría un Q&A extensivo (como ocurrió recientemente con problemas del scope en `handleObsTypeChange`). Estimación: **A 1 mes de refactorización estabilizadora para versión 1.0 real.**

---

## 9. RECOMENDACIONES TÉCNICAS (PLAN ACCIONABLE)

**🟢 Prioridad Alta (Inmediata: Próximos 2-4 días)**
1. **Sanear la Migración UUID:** Definir un corte (Hard-Block). Limpiar el código backend de `legacy_id` por completo en los Schemas y forzar actualización del Front. Cortar todos los `is_int` inferidos.
2. **Apagar Debugger en `main.py`:** Remplazar el `debug_exception_handler` por un loggueo local interno asíncrono con Python Logger estándar. No retornar el Traceback al cliente en HTTP 500.

**🟡 Prioridad Media (Refactorización a 2-3 semanas)**
1. **Desacoplar Frontend UI:** Tomar el archivo gigante `index.html` y romperlo radicalmente usando, como mínimo, el patrón Módulos ES6 nativos (`<script type="module">`). Separar las capas de API Fetch de la lógica DOM.
2. **Consolidar Permisos (`auth.py`):** Eliminar el bypass salvaje de roles (`"SISTEMAS"` in `user_role_names`). Asignar los permisos base directamente en la tabla relacional DB para el rol de `SISTEMAS` para un control granular verdadero.

**⚪ Prioridad Baja (Deuda Técnica a Mediano Plazo)**
1. Mover configuraciones *hardcodeadas* de horas de receso a `BreakConfig` (tabla DB) y consultarlas dinámicamente, asegurando que las planillas históricas no cambien cuando varían los horarios.
2. Implementar Tests de Automatizados Core (pytest) evaluando la normalización MDM (`fuzzy_match` edge cases) para que un *deploy* no degenere la data.

## 3. Integridad y Coherencia Referencial de los Respaldos JSON

Se ejecutó un script de diagnóstico forense sobre los archivos JSON para verificar si la pérdida o cambio de IDs afectaba las relaciones. Los resultados son extraordinariamente positivos:
* **Docentes**: 190 UUIDs únicos en total.
* **Cursos**: 236 UUIDs únicos en total.
* **Aulas**: 153 UUIDs únicos en total.

  * *Referencias faltantes de Docentes en Lecciones*: **0**
  * *Referencias faltantes de Cursos en Lecciones*: **0**
  * *Referencias faltantes de Aulas en Lecciones*: **0**
* **Sesiones**: 8,402 sesiones de clase.
  * *Referencias faltantes de Lecciones en Sesiones*: **0**

> [!NOTE]
> Esto confirma que el set de respaldos JSON en `backup_json/` es una **copia fiel, consistente y de alta integridad** de la base de datos histórica en su estado post-migración a UUID. Su restauración es totalmente viable y segura.

---

## 4. Causa Raíz de la Anomalía

1. **Migración a UUID**: El sistema experimentó una evolución en el esquema pasando de IDs enteros secuenciales (`serial`) a identificadores únicos globales (`UUID`). Durante este proceso se realizó un respaldo completo del estado transicional en formato JSON serializado (en la carpeta `backup_json/`).
2. **Reinicio de Base de Datos**: Posteriormente, la base de datos activa fue reiniciada o recreada limpia para aplicar las nuevas restricciones de integridad referencial.
3. **Omisión de Restauración**: Se omitió la ejecución de un script de carga que re-insertara los 8,402 registros históricos mapeados con sus nuevos UUIDs. En su lugar, el sistema fue inicializado con datos semilla mínimos y una sola carga XML de prueba (`complex_mdm_test_v2.xml`), la cual generó los pocos docentes y registros semilla visibles actualmente.

---

## 5. Plan de Acción Forense para la Recuperación

Para solucionar este inconveniente de forma segura y restaurar el sistema a su estado real de producción sin romper la integridad del esquema, se propone el siguiente procedimiento:

1. **Generación de un Snapshot de Seguridad**: Crear una copia de respaldo en caliente de la base de datos PostgreSQL activa antes de realizar cualquier cambio destructivo.
2. **Desarrollo de un Script de Restauración Altamente Estructurado**:
   * Desactivar temporalmente los triggers/constraints de clave foránea o cargar las tablas en orden de dependencia jerárquica estricta.
   * Cargar las entidades independientes: `users`, `teachers`, `subjects`, `classes` preservando sus UUIDs originales del JSON.
   * Cargar las entidades dependientes de primer nivel: `lessons` (que enlazan `teachers`, `subjects`, `classes`).
   * Cargar las entidades dependientes de segundo nivel: `schedule_sessions` (que enlazan `lessons`).
   * Cargar las observaciones y reemplazos (`observations`).
   * Cargar el consolidado de planillas (`rpt_planilla`).
3. **Cruce y Vinculación XML**:
   * Crear un registro de carga XML histórico virtual en la tabla `xml_uploads` para asociar correctamente los registros históricos de `schedule_sessions` y `rpt_planilla` con un origen claro, satisfaciendo la restricción `xml_upload_id` recién integrada en conversaciones previas.
4. **Validación Forense Post-Restauración**:
   * Ejecutar la suite de pruebas unitarias del backend.
   * Verificar la visualización en el frontend mediante el navegador.

# Critical Business & Academic Flows

The following flows are the "spinal cord" of the system and must be protected by rigorous automated testing.

## 1. The Ingestion Pipeline (The "Golden Path")
**Sequence**: XML Upload → Parser → MDM Normalization → Session Generation → RPT Refresh.
**Risk**: A bug in the parser could corrupt the entire academic load for a semester.
**Validation**: Bit-identical output comparison of `rpt_planilla` records after XML ingestion.

## 2. The Payroll Logic (RPT)
**Sequence**: Fetch Sessions → Apply Observations (Faltas/Reemplazos) → Apply Recess Rules → Group by Teacher/Day.
**Risk**: Error in the 0.33 recess deduction or in the replacement teacher 100% credit transfer.
**Validation**: Mathematical verification against "Golden Datasets".

## 3. The Security Barrier (RBAC)
**Sequence**: User Login → JWT Generation → Dependency Injection `require_permission` → JSON Response.
**Risk**: A "DIRECCION" user performing "SISTEMAS" actions (e.g., deleting users).
**Validation**: Headless matrix testing for all roles vs all critical endpoints.

## 4. The MDM Integrity (Docentes)
**Sequence**: Fuzzy Match Search → Promotion from SinAsignar → Retrospective link to observations.
**Risk**: Creating duplicate teachers or losing history after a merge.
**Validation**: Unit tests for `normalize_name` and integration tests for `fusionar`.

# Data Flow Trace (End-to-End)

## Análisis del Ciclo de Vida del Dato (Caso: Docentes XML sin DNI)

### 1. Ingesta (XML → Persistencia DB)
* **Paso:** `XMLUploadService` recibe el archivo XML.
* **Procesamiento:** Se extraen las tarjetas, periodos y lecciones. Los docentes (`Teacher`) se insertan a la base de datos si no existen usando `get_or_create`.
* **Resultado:** Si un docente no tiene DNI en el XML, se guarda en la tabla `docentes` con `dni = NULL`. (Validado: Hay 22 docentes en este estado).
* **Veredicto de Flujo:** **PASÓ.** La persistencia es 100% correcta.

### 2. Consolidación Interna (DB → Repository)
* **Paso:** Módulos piden datos a los Repositorios.

# Resilience & High-Concurrency Governance

## 1. Concurrency Control (Advisory Locks)
- **Mechanism**: PostgreSQL Session-Level Advisory Locks (`pg_advisory_xact_lock`).
- **Scope**: All XML upload processes must acquire a lock with ID `1001`.
- **Behavior**: If an upload is running, concurrent requests will wait for the lock to release, ensuring sequential execution and preventing race conditions during entity mapping.

## 2. Idempotency (File Hashing)
- **Mechanism**: SHA-256 content hashing.
- **Rule**: Before processing, the system checks if a file with the same hash has already been `COMPLETED`.
- **Override**: If `overwrite=true` is passed, the system re-processes the file regardless of previous history.

## 3. Data Integrity (UPSERT)
- **Strategy**: `ON CONFLICT DO UPDATE` (Insert or Update).
- **Constraints**:
  - `schedule_sessions`: `(lesson_id, session_date, start_time)`
  - `rpt_planilla`: `(fecha_clase, docente, hora_inicio)`
- **Benefit**: Re-uploads or overlapping files do not create duplicates, but update existing records to the latest state.

## 4. Transactional Reliability
- **Unit of Work**: Each XML upload is a managed transaction.
- **Batching**: Large inserts are batched (1000 records) but wrapped in a high-level transaction that tracks `status` in `xml_uploads`.
- **States**: `PENDING` → `PROCESSING` → `COMPLETED` | `FAILED`.

## 5. Observability
- **Metrics**: Throughput (records/sec) is logged for every bulk operation.
- **Traceability**: All logs related to an upload are tagged with the `upload_id`.

## 6. Stress Testing
- **Procedure**: Periodically run `scripts/stress_test_xml.py` to ensure that advisory locks are correctly serializing requests and that UPSERT is preventing duplication under load.

# VONEX — ESTRATEGIA DE REGRESIÓN Y PARIDAD RPT

## 1. Protocolo "Bit-Identical Parity"

El objetivo de la estrategia de regresión es garantizar que cualquier modificación en el código no altere el cálculo académico validado.

### Principios:
- **Salida Determinística**: Los inputs (XML) deben producir siempre los mismos registros RPT.
- **Unidades de Medida**: El sistema opera en "Horas Académicas" (1 unidad = 50 min).
- **Consolidación**: El motor debe respetar la ventana de 20 min para fusiones de bloques.
- **Recesos**: La deducción de 0.33 debe ser exacta y única por docente/día.

## 2. Gestión de Golden Datasets

Ubicación: `tests/golden_datasets/`

- **`xml_simple.xml`**: Casos base de ingesta y mapeo.
- **`xml_recesos.xml`**: Casos complejos de transferencia de recesos y consolidación.
- **`xml_faltas_reemplazos.xml`** (Pendiente): Casos de impacto de observaciones.

### Proceso de Actualización:

1. Actualizar el XML correspondiente.
2. Ejecutar el test con `-s` para ver la nueva salida.
3. Actualizar las aserciones en `test_rpt_parity.py` solo tras validación manual del USER.

# Reporte: Corrección de Validación de Respuestas

## Origen del Problema
FastAPI se apoya en Pydantic para validar y documentar dinámicamente las entradas y salidas (Payloads) de la aplicación a través de Schemas.
El esquema `TeacherHoursItem` en `app/modules/docentes/schemas.py` es el responsable de dictaminar cómo se empaca la información del docente junto con la cantidad de sus horas, el cual es mandado al módulo de **RPT Planillas**.

El código original forzaba a que `dni` sea una cadena explícita.
```python
# ANTES
class TeacherHoursItem(BaseModel):
    teacher_id: UUID
    name: str
    dni: str # <-- ESTO BLOQUEABA A LOS DE XML
    total_hours: float
```

Al procesar la respuesta, cuando llegaba un docente válido en la DB pero con el campo `dni` en estado `NULL` (proveniente de un import XML donde dicho campo es incierto), Pydantic disparaba una excepción interna: `fastapi.exceptions.ResponseValidationError`.

Este error provocaba un HTTP 500 para toda la llamada.

## La Solución Quirúrgica
Se implementó un cambio "minimalista" modificando el contrato para indicar que el campo puede llegar nulo:
```python
# DESPUÉS
class TeacherHoursItem(BaseModel):
    teacher_id: UUID
    name: str
    dni: Optional[str] = None # <-- PERMISIVO Y SEGURO
    total_hours: float
```

## Validación
Los logs del sistema tras la corrección evidencian que el crash desapareció y que la API ahora envía el payload de forma resiliente:

```log

INFO:  [API PAYLOAD NORMALIZED] successfully returning 15 teacher hours items.
```

# Análisis de Causa Raíz (Root Cause Analysis)

## El Problema Fundamental
Los datos históricos (Docentes del XML y registros de sesiones y planillas) **SÍ EXISTEN** íntegros en PostgreSQL, pero no se visualizan a través del frontend. Anteriormente se detectaron filtros DNI en las capas Repository, pero incluso después de quitarlos y reiniciar la DB, el flujo seguía deteniéndose.

## Las Tres Capas de la Causa Raíz

### 1. El Bloqueador Fuerte: Tipado Estricto (Pydantic ValidationError)
Al quitar los filtros en `get_active_teachers_for_rpt`, los docentes importados por XML sin DNI comenzaron a pasar hacia la capa del Serializador de la API. 
FastAPI usa el esquema `TeacherHoursItem` que definía obligatoriamente `dni: str`. Al recibir un `None` de la base de datos, Pydantic lanza una excepción `ResponseValidationError`.
* **Consecuencia:** FastAPI corta el Request devolviendo un Código HTTP 500 y no entrega NADA de la lista al Frontend. RPT Planillas colapsa por completo.

### 2. El Culpable Silencioso: Fallback de Nombres en el Frontend (Horarios)
En el módulo de Horarios, `/api/schedule/classes` y `/api/schedule/teachers` no colapsan, pero el frontend tiene lógica frágil. 
```javascript
displayName = displayName.replace(/^,|,$/g, '').trim();
if (!displayName || displayName === '.') {
    displayName = "Docente Sin Nombre Asignado";
}
```
Si un objeto se vuelve nulo o no se empareja correctamente, se agrupa todo bajo falsos nombres, o los modales de select impiden seleccionar el ítem si carece de un id explícito procesable.

### 3. El Bloqueador de UX: Reglas de Negocio en Javascript
Incluso viendo los docentes en la Maestra (donde ya logramos que aparezcan), el archivo `docentes_mgmt.js` impone reglas duras:
```javascript
if (!body.dni || body.dni.trim() === "") { alert("El DNI es obligatorio"); }
```
Esto fuerza a los usuarios a pensar que el docente está "corrupto" o que el flujo no sirve porque no pueden editarlo o guardarlo.

## Conclusión Técnica
La persistencia de la BD es perfecta. La consulta SQL es perfecta. El cuello de botella masivo es un desajuste entre los Contratos API (Pydantic que asume DNI obligatorio) frente a la realidad del Dominio (XMLs donde los profesores frecuentemente vienen sin DNI). El Frontend está reaccionando a estos crashes interrumpiendo el dibujado del DOM.

# Walkthrough: Resolución Quirúrgica DNI NULL

## Resumen del Ajuste
Se ejecutó la solución quirúrgica acordada para permitir que la lógica de negocio acepte profesores válidos, importados históricamente o de XMLs, que aún no cuentan con un número de DNI. 

A pesar de que el DNI suele considerarse un dato obligatorio en la etapa de contratación, el sistema no debe fallar internamente ni ocultar los datos de las horas trabajadas cuando esta información técnica falte.

## Cambios Realizados

### 1. El Quiebre en la Consolidación (Backend)
- **Problema:** En `app/modules/docentes/schemas.py`, el esquema de validación `TeacherHoursItem` tenía la obligación estricta de que el DNI fuera de tipo `str`. Cuando el ORM entregaba un valor `None` (NULL de PostgreSQL), el motor lanzaba un `ResponseValidationError` y botaba toda la petición con un error 500.
- **Ajuste:** Se volvió tolerante al cambiar de `dni: str` a `dni: Optional[str] = None`. 
- **Validación:** Se agregó trazabilidad con los logs `[TEACHER NULL DNI ACCEPTED]` y `[API PAYLOAD NORMALIZED]`.
- **Resultado:** ¡HTTP 200 OK! El RPT de planillas vuelve a cargar.

### 2. El Bloqueo del Usuario (Frontend)
- **Problema:** En `prototype/js/docentes_mgmt.js`, existía una validación que bloqueaba forzosamente la creación de cualquier Teacher en caso no se ingresara un DNI. Pero más grave aún, si un usuario ingresaba a *editar* a un profesor que provenía de XML (el cual no tenía DNI) para actualizar su nombre, el JS también bloqueaba esa operación.
- **Ajuste:** La obligación se delegó **exclusivamente** a la *creación manual* (`isManualCreation`), permitiendo guardar los registros históricos actualizados o provenientes del importador XML sin trabas.
- **Validación:** Logs introducidos para trazar `[NULL DNI FLOW ENABLED]`.

- **Migration:** Executed a surgical migration that flagged "SUPERADMIN" and "SISTEMAS" as protected roles.

### 2. Backend Auth Logic (Hybrid Transition)
Updated `require_permission` in `app/dependencies/auth.py` to implement a two-tier bypass:
- **Tier 1 (Structural):** If any of the user's roles have `is_protected == True`, access is granted immediately.
- **Tier 2 (Legacy):** If no protected role is found, it falls back to checking for the string `"SUPERADMIN"`.

### 3. Role Protection Service
Modified `app/modules/usuarios/service.py` to prevent accidental deletion or renaming of critical roles.
- **Enforcement:** Both `update_role` and `delete_role` now check the `is_protected` flag and a hardcoded critical list (`SUPERADMIN`, `ADMIN`, `DOCENTE`, etc.).
- **Trazabilidad:** Unauthorized attempts are logged with `[PROTECTED ROLE BLOCKED]`.

### 4. Frontend SPA Alignment
Updated `js/api.js` to support the new security model.
- **Bypass Priority:** The `hasPermission` function now prioritizes the `is_protected` flag returned by the API.
- **UX Consistency:** The legacy fallback ensures that existing sessions remain functional during the rollout.
